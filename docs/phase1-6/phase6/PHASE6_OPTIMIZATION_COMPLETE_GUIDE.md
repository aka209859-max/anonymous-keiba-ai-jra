# 🎯 Phase 6運用最適化 完全ガイド

## 📋 現状の問題点

ユーザーから報告された問題:

1. **処理時間が長すぎる**: PC-KEIBAで「データ登録」ボタンを押すと、保有している15年分すべてのJRDBファイルを処理し、**2時間30分**かかる
2. **不要なファイルも処理**: JRDBから提供される15種類のファイルすべてを処理しているが、Phase 6予測に必要なのは **4ファイルのみ**

---

## 🎯 目標

**週次運用で実現したいこと**:
- 処理時間: **2時間30分 → 5分以内** (97%削減)
- 登録対象: **15ファイル → 4ファイル** (KYI/CYB/JOA/SEDのみ)
- 登録期間: **15年分 → 直近30日分** (差分登録)
- 週次作業時間: **合計10分以内** (現在は2.5時間)

---

## 📊 Phase 6予測システムで必要なJRDBファイル

### ✅ 必須ファイル (4種類)

| ファイル | 内容 | Phase 6での用途 | 重要度 |
|:--------|:----|:--------------|:------|
| **KYI** | 競走馬基本情報・IDM・馬体重・オッズ | 馬の基本属性・JRDB独自指標 | ⭐⭐⭐⭐⭐ |
| **CYB** | 調教タイム・調教コース・調教指数 | 調教状態・追い切りタイム | ⭐⭐⭐⭐⭐ |
| **JOA** | 調教師・騎手コメント・注目馬情報 | コメント分析・注目度ランク | ⭐⭐⭐⭐ |
| **SED** | 確定着順・タイム・上がり3ハロン | レース後の検証・再学習用 | ⭐⭐⭐ |

### ❌ 不要ファイル (11種類) - 登録スキップ可能

BAC, CHA, KAB, OZ, OW, OU, OT, UKC, KKA, ZKB, ZED

**理由**:
- Phase 6予測では使用しない
- KYI/CYB/JOA で代替可能
- 処理時間の73%を占めている

---

## 🔧 最適化アプローチ (3つの軸)

### 1. **登録対象期間の限定**
```
現状: 2010年～2025年 (15年分) を毎回処理
↓
最適化: 直近30日のみを差分登録
```

### 2. **ファイル種別の限定**
```
現状: 15種類すべてを処理
↓
最適化: KYI, CYB, JOA, SED の4種類のみ
```

### 3. **差分登録モードの有効化**
```
現状: 全データ再登録 (既存データを削除して再登録)
↓
最適化: 差分登録モード (新規データのみ追加)
```

### ステップ1: PC-KEIBAの設定確認

1. PC-KEIBAを起動
2. ツールバー「**データ (D)**」→「**外部データ登録**」をクリック
3. 以下の項目を確認:
   - [ ] 「**設定**」「**オプション**」「**詳細設定**」等のボタンの有無
   - [ ] 「**登録対象期間**」の設定項目 (全期間 / 直近N日 / 指定期間)
   - [ ] 「**差分登録モード**」の ON/OFF スイッチ
   - [ ] 「**ファイル種別**」の選択画面 (チェックボックス等)

**📸 スクリーンショット取得**: 設定画面全体を撮影して報告

### ステップ2: 設定ファイル (XML) の確認

以下のフォルダを検索:
```
C:\Program Files\PC-KEIBA\
C:\Users\<username>\Documents\PC-KEIBA\
C:\Users\<username>\AppData\Roaming\PC-KEIBA\
```

検索対象ファイル名:
- `jrdb*.xml`
- `external_data*.xml`
- `config*.xml`

見つかったXMLファイルをテキストエディタで開き、以下の記述を探す:
```xml
<RegistrationConfig>
  <TargetPeriod>全期間</TargetPeriod>  <!-- ← ここを変更したい -->
  <IncrementalMode>false</IncrementalMode>  <!-- ← これをtrueにしたい -->
  <FileTypes>
    <File>KYI</File>
    <File>CYB</File>
    <!-- ... -->
  </FileTypes>
</RegistrationConfig>
```

### ステップ5: PostgreSQL データ検証

各パターン実行後、以下のSQLで登録結果を確認:

```sql
-- 1. 当日分のKYIデータが正常登録されているか確認
SELECT COUNT(*) AS today_count
FROM jrd_kyi
WHERE race_shikonen LIKE '260221%';
-- 期待値: 150～200件

-- 2. 4テーブルの整合性確認
SELECT 
  (SELECT COUNT(*) FROM jrd_kyi WHERE race_shikonen LIKE '260221%') AS kyi,
  (SELECT COUNT(*) FROM jrd_cyb WHERE race_shikonen LIKE '260221%') AS cyb,
  (SELECT COUNT(*) FROM jrd_joa WHERE race_shikonen LIKE '260221%') AS joa,
  (SELECT COUNT(*) FROM jrd_sed WHERE race_shikonen LIKE '260221%') AS sed;
-- 期待値: 4つとも同じ件数 (例: 158件ずつ)

-- 3. 過去データが削除されていないことを確認
SELECT COUNT(*) AS total_records,
       MIN(race_shikonen) AS oldest,
       MAX(race_shikonen) AS newest
FROM jrd_kyi;
-- 期待値: total_records = 約48万件 (2010～2025年)
```
